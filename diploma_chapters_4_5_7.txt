4 Разработка базы данных и пользовательского интерфейса

4.1 ER-диаграмма базы данных

В качестве системы управления данными в приложении используется облачная NoSQL-база данных Cloud Firestore от Google Firebase. В отличие от реляционных СУБД, Firestore хранит данные в виде коллекций документов, где каждый документ представляет собой набор пар «ключ — значение». Вложенные коллекции (subcollections) позволяют организовывать иерархические структуры данных без необходимости создания связующих таблиц.

Структура базы данных включает пять основных коллекций верхнего уровня и две вложенные подколлекции:

Коллекция users хранит профили всех зарегистрированных пользователей системы — как студентов, так и администраторов. Каждый документ идентифицируется уникальным идентификатором, совпадающим с uid пользователя в Firebase Authentication, что обеспечивает прямую связь между учётной записью и профилем. Документ содержит следующие поля: uid (String) — уникальный идентификатор пользователя; email (String) — адрес электронной почты; role (String) — роль пользователя, принимает значения «student» или «admin»; name (String) — отображаемое имя; points (Int) — текущее количество накопленных баллов; snakeDailyPoints (Int) — количество баллов, заработанных в мини-игре «Змейка» за текущий день; snakeDailyDate (String) — дата последнего обновления дневного лимита в формате «ГГГГ-ММ-ДД».

Подколлекция completedTasks, вложенная в документ пользователя по пути users/{uid}/completedTasks, хранит записи о выполненных заданиях конкретного студента. Каждый документ содержит: taskId (String) — идентификатор задания; title (String) — название задания; points (Int) — количество полученных баллов; completedAt (Long) — временная метка подтверждения выполнения.

Подколлекция redemptions, расположенная по пути users/{uid}/redemptions, содержит историю погашения наград студентом. Поля: rewardId (String) — идентификатор награды; rewardTitle (String) — название награды; rewardPoints (Int) — стоимость в баллах на момент погашения; timestamp (Long) — временная метка операции.

Коллекция tasks хранит все задания, созданные администраторами. Поля документа: title (String) — название задания; description (String) — подробное описание; points (Int) — количество баллов за выполнение; deadline (Long) — дедлайн в виде временной метки (Unix timestamp); при значении 0 дедлайн не установлен.

Коллекция taskSubmissions содержит заявки студентов на подтверждение выполнения заданий. Это центральный элемент рабочего процесса «выполнение → проверка → одобрение». Поля: studentId (String) — uid студента; studentName (String) — имя студента; taskId (String) — идентификатор задания; taskTitle (String) — название задания; points (Int) — баллы за задание; submittedAt (Long) — временная метка подачи заявки; status (String) — статус, принимает значения «pending» (ожидает проверки), «approved» (одобрено) или «rejected» (отклонено).

Коллекция rewards содержит каталог наград, доступных студентам для обмена на баллы. Поля: title (String) — название награды; description (String) — описание; points (Int) — стоимость в баллах; iconName (String) — кодовое имя иконки из набора Material Icons (gift, cafe, food, offer, trophy).

Коллекция notifications хранит уведомления, отправленные администратором всем студентам. Поля: title (String) — заголовок уведомления; body (String) — текст сообщения; sentAt (Long) — временная метка отправки.

Описанная структура обеспечивает эффективное разделение данных по ролям, минимальную избыточность и высокую скорость выборки за счёт встроенных возможностей индексирования Firestore. Все операции с баллами выполняются в рамках транзакций Firestore (runTransaction), что исключает состояние гонки и гарантирует согласованность данных при одновременном доступе нескольких пользователей.


4.2 Разработка клиентской части

Клиентская часть приложения разработана на языке Kotlin с использованием декларативного UI-фреймворка Jetpack Compose. Архитектура приложения построена по паттерну MVVM (Model-View-ViewModel), что обеспечивает строгое разделение ответственности между слоями: представление (Composable-функции), бизнес-логика (ViewModel) и данные (Repository + Firebase SDK).

Слой данных (Data Layer) реализован через набор репозиториев, каждый из которых инкапсулирует работу с конкретной коллекцией Firestore. AuthRepository отвечает за регистрацию и вход пользователей через Firebase Authentication, а также за сохранение профиля в коллекцию users при первой регистрации. TaskRepository обеспечивает CRUD-операции с заданиями: создание, получение списка, обновление и удаление. TaskSubmissionRepository реализует рабочий процесс подачи и проверки заявок на выполнение заданий. RewardRepository управляет каталогом наград. UserRepository отвечает за управление профилем пользователя, историей баллов, транзакциями списания баллов при погашении наград и логикой дневного лимита мини-игры.

Слой ViewModel использует StateFlow для хранения и обновления состояния экрана. Каждый экран имеет собственный ViewModel с классом состояния (UiState), который содержит все необходимые данные для отображения: флаги загрузки (isLoading), сообщения об ошибках (error), результаты запросов и промежуточные состояния форм. Изменения состояния автоматически вызывают перерисовку только тех Composable-функций, которые подписаны на изменившиеся данные, что обеспечивает высокую производительность интерфейса.

Слой представления (UI Layer) реализован как набор Composable-функций, разбитых по функциональным модулям. Навигация между экранами организована через Jetpack Navigation Compose с двумя вложенными NavHost: для роли студента и для роли администратора. После входа в систему маршрутизация выполняется на основе поля role в документе пользователя Firestore.

Приложение поддерживает два навигационных пространства. Для студентов нижняя панель навигации включает разделы: Главная, Задания, Игра, Награды и Профиль. Для администраторов нижняя панель включает разделы: Главная, Задания, Сдачи (проверка заявок), Статистика и Рейтинг; из главного экрана администратора доступны дополнительные разделы: Студенты и Награды.

Ключевые технические решения, применённые при разработке клиентской части:

Транзакционное начисление баллов. Все операции изменения баллов (начисление при одобрении заявки, списание при погашении награды, начисление в мини-игре) выполняются через Firestore runTransaction. Это атомарная операция: либо все изменения применяются одновременно, либо не применяется ни одно, что исключает рассогласование данных.

Реактивное обновление интерфейса. Для отображения профиля студента используются snapshot-слушатели Firestore (addSnapshotListener), которые автоматически уведомляют приложение о любых изменениях документа в реальном времени без необходимости явного обновления.

Ограничение дневного лимита игры. В мини-игре «Змейка» студент может заработать не более 25 баллов в сутки. Реализация использует поля snakeDailyPoints и snakeDailyDate в документе пользователя. При каждом начислении баллов транзакция сравнивает текущую дату с сохранённой и обнуляет счётчик при смене дня.

Защита от дублирования заявок. Перед созданием записи в коллекции taskSubmissions выполняется запрос, проверяющий наличие заявки от данного студента на данное задание со статусом «pending». Повторная подача блокируется на уровне репозитория с возвратом понятного пользователю сообщения об ошибке.

Push-уведомления реализованы через Firebase Cloud Messaging (FCM). Студенты автоматически подписываются на топик «students» при входе в систему. Администратор формирует и отправляет уведомления через панель управления приложения: сообщение публикуется в топик, и FCM доставляет его на устройства всех подписанных студентов. Обработка входящих уведомлений выполняется в классе MyFirebaseMessagingService, наследующем FirebaseMessagingService.


4.3 Разработка серверной части

Серверная часть приложения реализована полностью на платформе Firebase, без разработки собственного бэкенд-сервера. Такой подход обозначается термином «Backend as a Service» (BaaS) и позволяет сосредоточить усилия на разработке клиентского приложения, делегировав хранение данных, аутентификацию и доставку уведомлений готовой облачной инфраструктуре.

Firebase Authentication обеспечивает регистрацию и аутентификацию пользователей по методу email/password. При успешной регистрации Firebase автоматически генерирует уникальный идентификатор пользователя (uid), который используется в качестве ключа документа в коллекции users. Это создаёт однозначную связь между учётными данными и профилем без необходимости разработки отдельного механизма идентификации.

Cloud Firestore обеспечивает хранение всех структурированных данных приложения. Правила безопасности Firestore (Security Rules) определяют политику доступа к данным: студент может читать и записывать только в своей подколлекции, не имея доступа к данным других пользователей; администратор имеет расширенные права на чтение и запись в основные коллекции заданий и заявок.

Firebase Cloud Messaging (FCM) используется для отправки push-уведомлений. Топиковая модель доставки (topic messaging) позволяет отправить одно сообщение сразу всем подписанным устройствам без необходимости хранить список токенов устройств на стороне приложения.

Безопасность передачи данных обеспечивается протоколом HTTPS, который Firebase использует для всех соединений по умолчанию. Данные в базе хранятся в зашифрованном виде на серверах Google, что соответствует требованиям современных стандартов защиты информации.


4.4 Реализация пользовательского интерфейса

Пользовательский интерфейс реализован на основе дизайн-системы Material Design 3, адаптированной под фирменную стилистику приложения. Цветовая схема построена на градиентном фоне от #ECDFFF до #FEFBFF (светло-фиолетовый), акцентном цвете #7B61FF (индиго) и основном текстовом цвете #2D1B69 (тёмно-фиолетовый). Такое сочетание создаёт современный, лёгкий и привлекательный для студенческой аудитории интерфейс.

Экран приветствия предоставляет пользователю выбор между входом и регистрацией. Экраны авторизации содержат поля ввода email и пароля с встроенной валидацией. При регистрации новый студент выбирает отображаемое имя, которое сохраняется в профиле Firestore. После успешной аутентификации приложение проверяет роль пользователя и перенаправляет его в соответствующее навигационное пространство.

Главный экран студента (Home) отображает карточку профиля с именем, текущим количеством баллов и аватаром (инициалы пользователя). Ниже представлены карточки навигации к основным разделам и блок последних уведомлений от администратора.

Экран заданий (Tasks) выводит список доступных заданий в виде карточек с названием, описанием, количеством баллов и дедлайном. Выполненные задания визуально отмечены зелёной галочкой. Для ещё не выполненного задания доступна кнопка «Отправить на проверку», при нажатии которой создаётся запись в коллекции taskSubmissions. Реализован поиск по заданиям в режиме реального времени.

Экран наград (Rewards) отображает каталог наград с иконкой, названием, описанием и стоимостью в баллах. При нажатии на кнопку обмена выполняется транзакция: баллы списываются из профиля студента, и в подколлекции redemptions создаётся запись об обмене. Если баллов недостаточно, операция блокируется с отображением соответствующего сообщения.

Экран рейтинга (Rating / Leaderboard) выводит список всех студентов, отсортированный по убыванию баллов. Текущий пользователь выделяется цветом. Для каждого студента отображается порядковое место, имя и количество баллов.

Экран мини-игры реализует классическую игру «Змейка» с управлением через сенсорные кнопки направления. По завершении раунда подсчитываются очки и начисляются баллы (с учётом дневного лимита в 25 баллов). Игровое поле отрисовывается через Canvas API Compose.

Экран профиля студента отображает текущие данные пользователя: имя, email, баллы, историю выполненных заданий и историю погашенных наград. Предусмотрена возможность редактирования отображаемого имени.

Панель администратора начинается с главного экрана (Admin Home) с четырьмя статистическими карточками: количество студентов, заданий, средний балл и лидер. Из главного экрана доступны разделы управления наградами и рассылки уведомлений студентам.

Экран управления заданиями (Task Management) поддерживает полный CRUD-цикл: создание нового задания через модальную нижнюю панель (BottomSheet) с полями для названия, описания, баллов и дедлайна; редактирование и удаление существующих заданий. Реализован поиск по заданиям.

Экран проверки заявок (Submissions) отображает список заявок студентов в статусе «pending». Для каждой заявки администратор может нажать «Одобрить» — тогда баллы начисляются студенту атомарной транзакцией, — или «Отклонить» — статус заявки меняется на «rejected».

Экран студентов выводит список всех зарегистрированных студентов с их баллами. Нажатие на студента открывает его подробный профиль: имя, email, баллы, история выполненных заданий и история обмена наград.

Экран статистики (Statistics) представляет агрегированные данные: общее количество студентов, активных студентов, заданий, наград, суммарные и средние баллы, а также гистограмму распределения студентов по диапазонам баллов и список топ-студентов с горизонтальными прогресс-барами.

Экран управления наградами (Rewards) позволяет администратору создавать, редактировать и удалять награды каталога. Форма содержит поля названия, описания, стоимости и выбор иконки из пяти вариантов. Удаление сопровождается диалогом подтверждения.

5 Тестирование приложения

5.1 Виды и методы тестирования

Тестирование мобильного приложения является обязательным этапом разработки, позволяющим убедиться в корректности работы всех функциональных компонентов, стабильности пользовательского интерфейса и надёжности взаимодействия с серверной инфраструктурой Firebase. В рамках данного проекта применялись следующие виды тестирования: функциональное тестирование, ручное тестирование пользовательского сценария и тестирование граничных условий.

Функциональное тестирование направлено на проверку соответствия реализованных функций установленным требованиям. Для каждой ключевой функции составлен тест-кейс с описанием предусловий, шагов выполнения и ожидаемого результата.

Ручное тестирование проводилось непосредственно на физическом Android-устройстве. Приложение устанавливалось через Android Studio в режиме отладки. Тестирование охватывало все основные пользовательские сценарии для обеих ролей.

Тестирование граничных условий включало проверку поведения системы в нестандартных ситуациях: попытка зарегистрироваться с уже существующим email, отправка пустой формы, попытка обмена награды при недостаточном количестве баллов, повторная отправка заявки на уже отправленное задание, достижение дневного лимита баллов в мини-игре.


5.2 Тест-кейсы и результаты тестирования

Тест-кейс № 1. Регистрация нового пользователя.
Предусловия: приложение запущено, пользователь не авторизован.
Шаги: открыть экран регистрации; ввести корректный email; ввести пароль не менее 6 символов; ввести отображаемое имя; нажать «Зарегистрироваться».
Ожидаемый результат: создаётся учётная запись в Firebase Authentication; в коллекции users создаётся документ с uid, email, role = «student», name и points = 0; пользователь перенаправляется на главный экран студента.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 2. Регистрация с существующим email.
Предусловия: в Firebase уже зарегистрирован пользователь с данным email.
Шаги: ввести уже используемый email, пароль и имя; нажать «Зарегистрироваться».
Ожидаемый результат: отображается сообщение об ошибке «Пользователь с таким email уже существует».
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 3. Вход в систему.
Предусловия: пользователь зарегистрирован в системе.
Шаги: открыть экран входа; ввести email и пароль; нажать «Войти».
Ожидаемый результат: пользователь авторизован; в зависимости от роли (student/admin) выполняется переход на соответствующий главный экран.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 4. Отправка задания на проверку.
Предусловия: студент авторизован; в системе есть хотя бы одно задание; студент ранее не отправлял заявку на это задание.
Шаги: перейти в раздел «Задания»; нажать «Отправить на проверку» у выбранного задания.
Ожидаемый результат: в коллекции taskSubmissions создаётся документ со статусом «pending»; в интерфейсе задание помечается как отправленное.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 5. Повторная отправка задания.
Предусловия: студент уже отправил заявку на данное задание со статусом «pending».
Шаги: попытаться повторно нажать «Отправить на проверку» для того же задания.
Ожидаемый результат: отображается сообщение «Заявка уже отправлена»; новая запись в коллекции не создаётся.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 6. Одобрение заявки администратором.
Предусловия: администратор авторизован; в коллекции taskSubmissions есть заявка со статусом «pending».
Шаги: перейти в раздел «Проверка заявок»; нажать «Одобрить» у нужной заявки.
Ожидаемый результат: транзакцией Firestore баллы студента увеличиваются на указанное количество; в подколлекции completedTasks студента создаётся запись о выполненном задании; статус заявки меняется на «approved»; заявка исчезает из списка ожидающих.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 7. Отклонение заявки администратором.
Предусловия: администратор авторизован; есть заявка со статусом «pending».
Шаги: нажать «Отклонить» у выбранной заявки.
Ожидаемый результат: статус заявки меняется на «rejected»; баллы студента не изменяются; заявка исчезает из списка ожидающих.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 8. Обмен награды при достаточном количестве баллов.
Предусловия: студент авторизован; в каталоге есть награда; у студента баллов не меньше стоимости награды.
Шаги: перейти в раздел «Награды»; нажать кнопку обмена у выбранной награды; подтвердить действие.
Ожидаемый результат: транзакцией баллы студента уменьшаются на стоимость награды; в подколлекции redemptions создаётся запись; интерфейс обновляет отображение баллов.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 9. Обмен награды при недостаточном количестве баллов.
Предусловия: у студента баллов меньше, чем стоимость выбранной награды.
Шаги: нажать кнопку обмена у награды с высокой стоимостью.
Ожидаемый результат: транзакция завершается ошибкой «Недостаточно баллов»; баллы студента не изменяются; запись в redemptions не создаётся.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 10. Дневной лимит баллов в мини-игре.
Предусловия: студент авторизован; поле snakeDailyPoints в Firestore содержит значение 25 (лимит исчерпан на текущий день).
Шаги: запустить мини-игру «Змейка»; завершить раунд с любым счётом.
Ожидаемый результат: отображается сообщение о достижении дневного лимита; баллы за игровой раунд не начисляются.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 11. Создание задания администратором.
Предусловия: администратор авторизован.
Шаги: перейти в раздел «Задания»; нажать кнопку «+»; заполнить форму (название, описание, баллы, дедлайн); нажать «Создать».
Ожидаемый результат: в коллекции tasks создаётся новый документ; задание немедленно появляется в списке и становится доступным студентам.
Фактический результат: соответствует ожидаемому. Тест пройден.

Тест-кейс № 12. Отправка push-уведомления студентам.
Предусловия: администратор авторизован; хотя бы один студент подписан на топик «students».
Шаги: на главном экране администратора нажать «Написать студентам»; ввести заголовок и текст; нажать «Отправить всем».
Ожидаемый результат: Firebase Cloud Messaging доставляет уведомление на устройства студентов; на устройстве студента появляется системное уведомление с заголовком и текстом.
Фактический результат: соответствует ожидаемому. Тест пройден.

7 Управление проектом

7.1 Устав проекта

Название проекта: «Разработка мобильного приложения по учёту успеваемости студентов».

Основание для открытия проекта. Отсутствие доступного, специализированного цифрового инструмента для учёта внеаудиторной и аудиторной активности студентов на уровне кафедры. Существующие решения (Элжур, Habitica, Duolingo) не обеспечивают полного набора требуемых функций: гибкой настройки типов активности, роли администратора, физических вознаграждений и интеграции геймификации в контекст учебного процесса конкретного колледжа.

Цель проекта. Разработать и ввести в эксплуатацию мобильное Android-приложение с ролевой моделью (администратор / студент), автоматизирующее учёт активности студентов и повышающее их вовлечённость через систему баллов и геймификации.

Результат проекта. Работающее мобильное приложение, развёрнутое на Firebase, готовое к установке на Android-устройства и включающее: систему аутентификации с разграничением ролей; модуль управления заданиями; систему подачи и проверки заявок на выполнение заданий; каталог наград с функцией обмена баллов; мини-игру «Змейка» с дневным лимитом начисления баллов; экраны рейтинга, статистики и профиля; push-уведомления через Firebase Cloud Messaging.

Менеджер проекта: Чернышенко У.А.
Научный руководитель: Дубинин А.В.
Плановые сроки реализации: февраль — май 2026 года.

7.4 Заинтересованные стороны проекта

Студенты — основные конечные пользователи приложения. Их интересы: простой и привлекательный интерфейс, прозрачная система начисления баллов, справедливый рейтинг, реальные вознаграждения. Уровень влияния: высокий (пользовательское принятие определяет успех проекта).

Администраторы (преподаватели) — вторая группа пользователей, управляющая контентом. Их интересы: удобный инструмент добавления заданий, прозрачный мониторинг активности, возможность быстрой рассылки уведомлений. Уровень влияния: высокий (без администраторов система не наполняется контентом).

Научный руководитель — осуществляет контроль качества и академическое сопровождение. Его интересы: соответствие работы стандартам дипломного проекта, техническая состоятельность решения. Уровень влияния: средний.

Калининградский Бизнес-Колледж — потенциальный эксплуатант системы на уровне кафедры. Его интересы: снижение административной нагрузки на преподавателей, повышение вовлечённости студентов, отсутствие затрат на серверную инфраструктуру. Уровень влияния: средний.


7.5 Реестр рисков проекта

Риск 1. Технические сложности при работе с Firebase SDK.
Вероятность: средняя. Влияние: высокое.
Описание: несовместимость версий библиотек, изменения в API Firebase, ошибки конфигурации Gradle.
Меры по снижению: использование стабильной версии Firebase BOM (33.5.0), регулярная проверка официальной документации, поэтапная интеграция компонентов.

Риск 2. Ограничения бесплатного тарифа Firebase.
Вероятность: низкая. Влияние: среднее.
Описание: превышение лимитов на количество операций чтения/записи при масштабировании.
Меры по снижению: оптимизация запросов (использование snapshot-слушателей вместо повторных запросов), мониторинг потребления в консоли Firebase.

Риск 3. Низкая вовлечённость студентов.
Вероятность: средняя. Влияние: высокое.
Описание: студенты не будут активно использовать приложение, если задания неинтересны или не видят ценности в баллах.
Меры по снижению: привязка баллов к реальным вознаграждениям, создание разнообразных типов заданий, регулярная рассылка мотивирующих уведомлений.

Риск 4. Дефицит времени на разработку.
Вероятность: средняя. Влияние: высокое.
Описание: совмещение разработки с учёбой и написанием диплома может привести к нехватке времени.
Меры по снижению: расстановка приоритетов (MVP-подход: сначала ключевые функции, затем расширение), еженедельный контроль выполнения плана.

Риск 5. Ошибки в логике транзакций (некорректное начисление баллов).
Вероятность: низкая. Влияние: критическое.
Описание: гонка данных при одновременном изменении баллов с разных устройств.
Меры по снижению: все операции с баллами реализованы через атомарные транзакции Firestore (runTransaction), что исключает состояние гонки.

ЗАКЛЮЧЕНИЕ

В ходе выполнения дипломной работы было разработано мобильное Android-приложение «Система учёта активности студентов», предназначенное для повышения вовлечённости студентов в учебный и внеучебный процесс посредством механизмов геймификации.

В процессе работы были решены все поставленные задачи. Проведён анализ существующих аналогов (Элжур, Duolingo, Habitica), по результатам которого установлено, что ни одно из рассмотренных решений не обеспечивает полного набора требуемых функций: гибкой настройки типов активности, роли администратора с инструментами управления, системы проверки заявок и связи цифровых баллов с реальными вознаграждениями.

Сформулированы функциональные и нефункциональные требования к системе, выбран оптимальный технологический стек: язык программирования Kotlin, UI-фреймворк Jetpack Compose, архитектурный паттерн MVVM, облачная платформа Firebase (Authentication, Cloud Firestore, Cloud Messaging). Спроектирована структура базы данных NoSQL Firestore, включающая пять коллекций и две вложенные подколлекции.

Реализованы все запланированные функциональные модули: двухролевая система аутентификации, управление заданиями (CRUD), рабочий процесс подачи и проверки заявок на выполнение заданий с транзакционным начислением баллов, каталог наград с механизмом обмена баллов, экраны рейтинга и статистики, мини-игра «Змейка» с ограничением дневного заработка, система push-уведомлений через Firebase Cloud Messaging.

Проведено функциональное тестирование приложения: составлены и выполнены 12 тест-кейсов, охватывающих все ключевые пользовательские сценарии и граничные условия. Все тест-кейсы успешно пройдены, критических ошибок выявлено не было.

Практическая значимость разработанного приложения определяется его готовностью к внедрению на уровне кафедры без дополнительных затрат на серверную инфраструктуру: использование бесплатного тарифа Firebase обеспечивает работу системы для аудитории до нескольких сотен пользователей. Приложение снижает административную нагрузку на преподавателей, обеспечивает прозрачность системы поощрений и формирует позитивную культуру участия студентов в жизни учебного сообщества.

В качестве направлений дальнейшего развития проекта можно выделить: интеграцию с API социальной сети ВКонтакте для автоматического учёта активности; реализацию системы достижений и визуальной кастомизации профиля (звания, иконки); добавление модуля посещаемости с поддержкой QR-кодов; расширение аналитики для администратора; выпуск приложения в Google Play.
